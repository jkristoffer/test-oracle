# test-oracle — Technical Specification

**Version:** 0.1.0
**Status:** Draft
**Author:** Kris (Big Shine Singapore Pte Ltd)

---

## 1. Purpose

test-oracle is a standalone CLI tool that optimises test execution for AI coding agents. It ensures agents never run the full test suite during development — only the minimal set of tests affected by code changes get executed.

The tool is deterministic and LLM-agnostic. It handles detection, mapping, caching, and execution. It does NOT handle config generation or config updates — those are delegated to the AI agent via the skill layer.

---

## 2. Core Concepts

**Dependency Map:** A data store mapping each source file to the test IDs that executed code within it. Built from coverage data. Self-updates on every instrumented test run.

**Adapter:** An ecosystem-specific module that implements three operations: generate the dependency map, query the dependency map, and run selected tests. Each supported ecosystem (Node/TS, Python, Rust, etc.) provides its own adapter.

**Config (`.test-oracle.yml`):** Per-project configuration generated and maintained by an AI agent, not by the CLI. The CLI reads it but never writes it. See Section 7 for schema.

**Cache:** Content-hash-based store that maps a hash of (source files + test files + config) to a pass/fail result. Allows the orchestrator to skip execution entirely when nothing meaningful has changed.

---

## 3. CLI Interface

### Commands

```
test-oracle init
```
Runs the first full test suite with coverage instrumentation. Builds the initial dependency map from coverage output. Stores the map in `.test-oracle/map.db`. Requires `.test-oracle.yml` to already exist (generated by the agent).

```
test-oracle run [--files <path1> <path2> ...]
```
The primary command. Executes the optimised test pipeline (see Section 4). If `--files` is provided, uses those as the changed file set. Otherwise, resolves changed files from `git diff`.

```
test-oracle status
```
Reports: dependency map freshness (last updated timestamp, number of entries), cache state (number of cached results, hit rate), config validation result (valid/stale/errors).

```
test-oracle validate
```
Checks the current `.test-oracle.yml` against the project state. Reports issues: test patterns matching zero files, missing test commands, dependency map referencing deleted files, framework config files that changed since last init. Returns a structured result the agent can act on.

```
test-oracle reset
```
Clears the dependency map and cache. Forces a fresh `init` on next run.

## 4. Execution Pipeline (`test-oracle run`)

The pipeline executes sequentially. Each stage can terminate the pipeline early with a structured result.

### Stage 1: Detect

Resolve which source files changed.

- If `--files` flag is provided, use those paths directly.
- Otherwise, run `git diff --name-only` against the last known good state (stored in `.test-oracle/state`).
- Filter to only files matching source patterns defined in config (`source_patterns`).
- If zero files changed, return early: `{ status: "skip", reason: "no_changes" }`.

### Stage 2: Map

Look up changed files in the dependency map.

- For each changed file, retrieve the set of test IDs that have an execution dependency on it.
- Union all test IDs into a single deduplicated set.
- If the dependency map has no entry for a changed file (new file, never covered), fall back to convention matching: look for a test file with a matching name pattern.
- If zero tests are mapped, return early: `{ status: "skip", reason: "no_tests_mapped" }`.

### Stage 3: Cache Check

Hash the relevant files and check for a cached result.

- Compute hash: `sha256(content of changed source files + content of mapped test files + .test-oracle.yml)`.
- Look up hash in `.test-oracle/cache/`.
- If cache hit with status "pass", return early: `{ status: "pass", source: "cache", hash: "<hash>" }`.
- If cache hit with status "fail", do NOT return cached failure — re-run, as the agent likely made fixes.

### Stage 4: Static Checks

Run static analysis before any test execution.

- Execute each command listed in `static_checks` config array, scoped to changed files where possible.
- If any static check fails, return early with structured error. The test runner never starts.
- Result: `{ status: "fail", stage: "static", check: "tsc", error: "<output>" }`.

### Stage 5: Execute

Run the mapped tests.

- Invoke the test command via the ecosystem adapter, passing only the mapped test file paths.
- Enable fail-fast (`--bail`, `-x`, or equivalent per adapter).
- Capture structured output: pass/fail per test, error messages, duration.
- On completion, update the dependency map with fresh coverage data from this run.

### Stage 6: Result

Return structured JSON to the agent.

```json
{
  "status": "pass" | "fail",
  "stage": "detect" | "map" | "cache" | "static" | "execute",
  "source": "run" | "cache",
  "hash": "<content_hash>",
  "tests_run": ["auth.test.ts:login", "auth.test.ts:logout"],
  "tests_skipped": 38,
  "failed_test": null | "auth.test.ts:47",
  "error": null | "expected 401, got 200",
  "duration_ms": 1200,
  "map_updated": true,
  "static_checks_passed": true
}
```

On pass, store the hash in cache with status "pass".

## 5. Dependency Map

### Structure

```
source_file_path → [test_id, test_id, ...]
```

Where `test_id` is a framework-specific identifier (e.g., `tests/auth.test.ts`, `tests/test_auth.py::TestAuth::test_login`).

### Storage

SQLite database at `.test-oracle/map.db`. Two tables:

```sql
CREATE TABLE file_tests (
  source_path TEXT NOT NULL,
  test_id TEXT NOT NULL,
  last_updated INTEGER NOT NULL,
  PRIMARY KEY (source_path, test_id)
);

CREATE TABLE map_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- Keys: 'last_full_run', 'adapter', 'schema_version'
```

### Updates

The map updates incrementally after every test execution (Stage 5). Only the entries for files covered in the current run are refreshed. A full map rebuild happens only on `test-oracle init` or `test-oracle reset` followed by `init`.

### Staleness

If a source file exists but has no entry in the map, it means either:
- It's a new file never covered by tests (convention fallback applies).
- The map is stale and needs a full rebuild.

The `validate` command detects this and reports it.

## 6. Cache

### Structure

```
content_hash → { status, timestamp }
```

### Storage

Directory-based at `.test-oracle/cache/`. Each entry is a small JSON file named by hash.

### Hash Computation

```
hash = sha256(
  sorted(content of each changed source file) +
  sorted(content of each mapped test file) +
  content of .test-oracle.yml
)
```

### Eviction

Cache entries older than 7 days are evicted on each run. Configurable via `cache_ttl_days` in config.

### Rules

- Cache hits on "pass" → skip execution, return cached result.
- Cache hits on "fail" → ignore cache, re-run (agent likely made changes).
- Cache misses → run normally, store result on completion.

## 7. Config Schema (`.test-oracle.yml`)

This file is generated and maintained by the AI agent, not by the CLI. The CLI reads and validates it but never modifies it.

```yaml
# Required
ecosystem: node | python | rust | go
test_command: "vitest run"
coverage_command: "vitest run --coverage"
test_pattern: "src/**/*.test.ts"
source_patterns:
  - "src/**/*.ts"
  - "!src/**/*.test.ts"
  - "!src/**/*.d.ts"

# Optional
static_checks:
  - "tsc --noEmit"
  - "eslint --quiet"
fail_fast: true                    # Default: true
cache_ttl_days: 7                  # Default: 7
coverage_format: "json"            # Default: adapter-specific
convention_map:                    # Fallback when map has no entry
  pattern: "{name}.test.{ext}"     # How test files relate to source files

# Informational (for agent context, not used by CLI)
notes: |
  Integration tests in tests/integration/ require
  the docker compose stack running.
  Auth module tests depend on JWT_SECRET env var.
module_boundaries:
  - scope: "src/auth/**"
    tests:
      - "src/auth/**/*.test.ts"
      - "tests/integration/auth.test.ts"
```

### Validation Rules

The `validate` command checks:

1. `ecosystem` matches a known adapter.
2. `test_command` is resolvable (binary exists on PATH).
3. `test_pattern` matches at least one file.
4. `source_patterns` match at least one file.
5. `static_checks` commands are resolvable.
6. Dependency map entries reference files that still exist.

Validation returns a structured result:

```json
{
  "valid": false,
  "errors": [
    { "field": "test_pattern", "message": "Pattern matches 0 files" }
  ],
  "warnings": [
    { "field": "map", "message": "12 entries reference deleted files" }
  ]
}
```

## 8. Adapter Interface

Each ecosystem adapter implements:

```typescript
interface TestAdapter {
  /** Run full suite with coverage, return raw coverage data */
  generateMap(config: Config): Promise<CoverageData>;

  /** Given a list of source files, return mapped test IDs */
  queryMap(db: MapDB, changedFiles: string[]): Promise<string[]>;

  /** Run specific tests by ID with fail-fast, return structured results */
  runTests(testIds: string[], config: Config): Promise<TestResult[]>;

  /** Parse coverage output into the standard dependency format */
  parseCoverage(raw: CoverageData): Promise<FileDependency[]>;
}
```

### Planned Adapters

| Ecosystem | Test Framework | Coverage Tool | Priority |
|-----------|---------------|---------------|----------|
| Node/TS   | vitest, jest  | c8, istanbul  | P0       |
| Python    | pytest        | coverage.py   | P1       |
| Rust      | cargo test    | tarpaulin     | P2       |
| Go        | go test       | go cover      | P3       |

## 9. Skill Layer Integration

The CLI is the deterministic engine. The skill layer (SKILL.md for Claude Code / Codex) handles the intelligent parts:

### What the skill tells the agent:

1. **When starting a new project:** Run `test-oracle validate`. If no `.test-oracle.yml` exists, analyze the project and generate one. Then run `test-oracle init`.

2. **When running tests:** Never call the test framework directly. Always use `test-oracle run`. Pass `--files` with the files you just changed.

3. **When `test-oracle validate` reports errors:** Read the validation output, analyze the project, and update `.test-oracle.yml` to fix the issues. Then re-run `test-oracle init` if the dependency map is stale.

4. **When interpreting results:** Parse the structured JSON. If `status: "fail"` and `stage: "static"`, fix the type/lint error before looking at test failures. If `stage: "execute"`, use `failed_test` and `error` to guide the fix.

### What the skill does NOT do:

- It does not contain test orchestration logic.
- It does not know about adapters or coverage formats.
- It only knows the CLI commands and how to generate/update the config file.

## 10. File Structure

```
.test-oracle/
├── map.db              # SQLite dependency map
├── cache/              # Hash-based result cache
│   ├── <hash>.json
│   └── ...
└── state               # Last known good commit/state
.test-oracle.yml        # Project config (agent-managed)
```

Add `.test-oracle/cache/` to `.gitignore`. The `map.db` and `state` files should be committed or stored as CI artifacts for team workflows.

---

## 11. Error Handling

All errors return structured JSON with an `error` field. The CLI never exits with unstructured stderr output.

| Scenario | Behaviour |
|----------|-----------|
| No `.test-oracle.yml` found | `{ error: "no_config", message: "Run test-oracle validate to check setup" }` |
| No `map.db` found | `{ error: "no_map", message: "Run test-oracle init to build dependency map" }` |
| Adapter not found for ecosystem | `{ error: "unknown_ecosystem", message: "No adapter for: <value>" }` |
| Test command not found on PATH | `{ error: "command_not_found", message: "<command> not resolvable" }` |
| Coverage output unparseable | `{ error: "coverage_parse_error", message: "<details>" }` |
| Config validation fails | Returns validation result (see Section 7), does not run tests |

## 12. Constraints and Non-Goals

### Constraints

- All output is structured JSON. No unstructured terminal output in normal operation.
- The CLI is stateless between invocations. All state lives in `.test-oracle/`.
- Config is read-only from the CLI's perspective. Only the agent writes it.
- The dependency map is the source of truth for test selection. Convention fallback is a safety net, not a primary mechanism.

### Non-Goals

- **Test generation.** test-oracle selects and runs tests. It does not write them.
- **CI/CD integration.** The CLI can be called from CI, but it does not manage pipelines, deploy artifacts, or integrate with specific CI platforms.
- **Multi-language monorepos in v0.** First version targets a single ecosystem per project. Monorepo support with per-package adapters is a future consideration.
- **Distributed caching.** Cache is local-only in v0. Shared/remote cache is a future consideration for team workflows.
- **Config generation.** The CLI validates config but never generates or modifies it. That responsibility belongs to the skill layer.

## 13. Build Order

1. **v0.1 — Node/TS adapter only.** Detector + Mapper + Cache + Runner for vitest/jest with c8/istanbul coverage. Single ecosystem, single project. Prove the pipeline works end-to-end.

2. **v0.2 — Validation and self-healing loop.** `validate` command, structured error reporting, stale map detection. This enables the agent-driven config update workflow.

3. **v0.3 — Python adapter.** pytest + coverage.py. Validates the adapter interface is truly ecosystem-agnostic.

4. **v0.4 — Claude Code skill.** SKILL.md that teaches the agent when and how to use test-oracle. Tested on a real project in the AI Factory pipeline.

5. **Future — Additional adapters, monorepo support, shared cache, CI artifact persistence.**

## 14. Config Generation Prompts

The CLI never generates or modifies `.test-oracle.yml`. Instead, two prompts guide the AI agent to perform config generation and updates. These prompts are referenced by the skill layer and executed by the agent (Claude Code, Codex, or equivalent).

---

### 14.1 Init Prompt

**Trigger:** No `.test-oracle.yml` exists in the project root, or `test-oracle validate` returns `{ error: "no_config" }`.

**Input (what the agent reads):**

| Source | Purpose |
|--------|---------|
| `package.json` / `pyproject.toml` / `Cargo.toml` | Detect ecosystem, test framework, dependencies |
| Test framework config (`vitest.config.ts`, `jest.config.js`, `pytest.ini`, `setup.cfg`) | Detect test command, coverage settings, custom patterns |
| `tsconfig.json` / `mypy.ini` / equivalent | Identify available static checks |
| Directory structure (top 3 levels) | Identify source and test file locations, naming conventions |
| `.eslintrc` / `.prettierrc` / linter configs | Identify additional static checks |
| Existing CI config (`.github/workflows/`, `.gitlab-ci.yml`) if present | Extract existing test commands and environment requirements as reference |

**Analysis the agent performs:**

1. Identify the ecosystem (node, python, rust, go) from project manifest files.
2. Identify the test framework and its invocation command.
3. Identify the coverage command or determine how to enable coverage for the detected framework.
4. Determine the coverage output format (json, lcov, etc.) supported by the framework.
5. Scan the directory structure to establish `test_pattern` and `source_patterns` with appropriate exclusions (e.g., type definitions, config files, build output).
6. Identify available static checks from installed tooling and their configs.
7. Detect any `convention_map` pattern by comparing source and test file naming (e.g., `auth.ts` → `auth.test.ts`, `auth.py` → `test_auth.py`).
8. Identify `module_boundaries` by scanning for distinct modules/packages with co-located or dedicated test directories.
9. Note any environment dependencies, required services, or setup steps in `notes`.

**Output:** A valid `.test-oracle.yml` written to the project root, conforming to the schema in Section 7.

**Post-generation step:** The agent runs `test-oracle validate` to verify the generated config. If validation fails, the agent fixes the config based on the validation output before proceeding to `test-oracle init`.

---

### 14.2 Update Prompt

**Trigger:** `test-oracle validate` returns `{ valid: false }` with errors or warnings, OR `test-oracle run` returns `{ error: "command_not_found" }` or similar config-related errors.

**Input (what the agent receives):**

| Source | Purpose |
|--------|---------|
| Current `.test-oracle.yml` | The existing config to patch |
| Validation output (from `test-oracle validate`) | Specific errors and warnings to address |
| `git diff` of config-relevant files since last successful run | What changed in the project that may have caused staleness |
| Same project files as Init Prompt | Current project state for re-analysis |

**Analysis the agent performs:**

1. Parse the validation errors and warnings to understand what's broken.
2. For each error, determine the fix:
   - `test_pattern matches 0 files` → Re-scan directory structure, update pattern.
   - `command_not_found` → Check if framework changed, update `test_command` / `coverage_command`.
   - `unknown_ecosystem` → Re-detect ecosystem from manifest files.
   - `map entries reference deleted files` → Not a config issue; recommend `test-oracle init` to rebuild the map.
3. For each warning, determine if action is needed or if it's informational.
4. Check if new static checks should be added (e.g., new linter installed since last config generation).
5. Check if `module_boundaries` need updating based on new directories or moved files.
6. Update `notes` if environment dependencies changed.

**Output:** A patched `.test-oracle.yml` written to the project root. Only the fields that need changing are modified — the agent does not regenerate the entire file.

**Post-update step:** The agent runs `test-oracle validate` again to confirm the fix. If the map is stale (deleted file references), the agent runs `test-oracle init` to rebuild it.

---

### 14.3 Prompt Boundaries

These prompts define what the agent does, not how the CLI works. The separation is strict:

| Responsibility | Owner |
|----------------|-------|
| Analyzing project structure | Agent (via prompts) |
| Generating `.test-oracle.yml` | Agent (via prompts) |
| Updating `.test-oracle.yml` | Agent (via prompts) |
| Validating `.test-oracle.yml` | CLI (`test-oracle validate`) |
| Flagging config staleness | CLI (`test-oracle validate`, `test-oracle run`) |
| Triggering the agent to update config | Skill layer (references these prompts) |
| All test detection, mapping, caching, execution | CLI |

The CLI is the trigger. The prompts are the intelligence. The skill layer is the glue that connects them.
